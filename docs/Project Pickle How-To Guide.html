Project Pickle How-To Guide

A friendly guide for setting up, updating, or upgrading CFI Access inventory databases.

Developed by the BIA Division of Forestry, Branch of Forest Inventory and Planning.

Author: Christopher LaCroix, USDI BIA Division of Forestry, Branch of Forest Inventory and Planning

Guide last reviewed: July 22, 2026 | Project Pickle app version reviewed: 2.1.4

Maintenance note: whenever Project Pickle is updated, review this guide before sharing the app. Check that tab names, button names, required entries, and troubleshooting advice still match the app.

What Project Pickle Does

Project Pickle builds CFI databases for new projects, updates tables in existing projects, and upgrades old CFI databases, tables and queries included, to the latest standard database with the click of a button.

Project Pickle is built to reduce repetitive database setup work, especially setup steps that are easy to miss by hand.

Version 2.1.4 includes the manual Reports designer, complete AppColumns/AppColumnCodes copy, an independent upgrade-only archive of the old AppColumns setup, completed report-table copy, two independent upgrade-only Processing imports, a three-table Processing compatibility fallback for older databases, automatic report spacing before Remarks fields, and an upgrade-only option to copy existing _prjCFIDEER_ queries instead of rebuilding them. The setup order is: Project, Periods, AppColumns, Upgrade Mapping, Plots, Get Field Order, Reports, CFIDEER Queries, Plot Assignments, Tree Assignments, Processing, Build, and AI Help.

  • It copies the selected database first and changes only the output copy.
  • It can set up the project, region, agency, reservation, and measurement periods.
  • It can import plots from a spreadsheet when plot data is ready.
  • It can create plot assignment records and upgrade-only tree assignment records.
  • It can review standard AppColumns and add new custom AppColumns.
  • It creates AppColumnsArchive in the saved project database before changing the working AppColumns setup.
  • In Upgrade Project mode, it can independently archive the old database's AppColumns and AppColumnCodes as zAppColumns and zAppColumnsCodes.
  • The old AppColumns archive option does not replace or change the working AppColumns and AppColumnCodes tables.
  • It provides a manual Reports tab for arranging ReportColumns, editing legacy formats, and building exact fixed-width ReportHeaders.
  • It can automatically add active ReportVisible Plot, Tree, and Regen custom fields to report layouts.
  • It automatically preserves RemarksTree as TEXT,40,5.
  • It automatically adds a trailing separator after the final custom Plot, Tree, or Regen field immediately before the required Remarks field.
  • In Upgrade Project mode, it can copy completed ReportHeaders and ReportColumns from the old database.
  • In Upgrade Project or Existing Project mode, it can completely replace working AppColumns and AppColumnCodes from the Existing/old project database.
  • In Upgrade Project mode, it can copy completed Processing setup tables from the old database.
  • When the old SpeciesMerchantabilityStandards table or its three required Yes/No fields are missing, it warns the user, copies the other three compatible Processing setup tables, and keeps SpeciesMerchantabilityStandards from the current master database version.
  • In Upgrade Project mode, it can independently copy PlotSummaries, TreeCalculations, and RegenCalculations from the old database.
  • It can update validation rules and the Access application title.
  • It can rebuild CFIDEER get and update queries when the current AppColumns and Get Field Order should be used.
  • In Upgrade Project mode, it can instead copy existing saved _prjCFIDEER_ queries from the old database.
  • It saves a detailed text run log beside the saved project database.
  • It includes an optional AI Help tab for setup questions.

Important safety rule: Project Pickle should not edit your master template or original existing project database. It makes an output copy and edits that copy. Still, always choose an output file name that is clearly different from the input database files.

How To Open The App

  1. Open the Project Pickle folder.
  2. Double-click Run Project Pickle.vbs.
  3. The app should open with the pickle icon.
  4. If needed, run the root-folder Run-ProjectPickle-32bit.bat for troubleshooting.

The main Project Pickle folder is intentionally simple: it shows a share-safe root launcher, the shortcut, and the Project Pickle App Files support folder. Run Project Pickle.vbs and the root Run-ProjectPickle-32bit.bat use relative paths, so they continue to work when the whole Project Pickle folder is shared with team members or moved to another location. The shortcut is still included for convenience on this local copy.

Do not open the internal app script directly. Some computers block unsigned PowerShell script files. The shortcut and Run-ProjectPickle-32bit.bat use the Project Pickle launcher so the app can start without that unsigned-script popup.

Start Here: Pick The Right Mode

The first choice on the Project tab is Mode. Pick the mode that matches the kind of database work you are doing.

Mode Use It When Database Box Or Boxes To Use
New Project A new CFI project with no prior inventory. Builds out the database for the first measurement using the current master database version. Use Current master database version. The Existing/old project database box is greyed out because there is no prior project database.
Upgrade Project

Any project using an older database version must go through the upgrade process. This happens in two cases:

  • Inactive database: Older databases stored in the Completed Crosswalk folder or project folder, usually previous projects or those waiting for remeasurement, must be upgraded to the current master version when remeasurement begins.
  • Database used externally: If a database was sent to a tribe for data collection or analysis and returns after the master version has changed, it must be migrated to the current version before processing or running analyses.
 Use both boxes: choose the new blank/current structure in Current master database version, then choose the old project database in Existing/old project database.
Existing Project Once a database is upgraded it might move into the Existing Project mode. The database structure and period rows are already built. Use this mode for adding new field data items or additional plots, then reviewing Get Field Order, CFIDEER queries, assignments, and Build. Use Existing/old project database for the current-format project database that needs updates. The current master database version box is greyed out.

Quick rule: choose New Project when there is no prior project database, Upgrade Project when the old database structure needs to be converted into the current master database version or a new remeasurement period must be added during upgrade, and Existing Project when the database is already in the current format and only needs fields, plots, query order, queries, assignments, reports, or validation refreshed.

The Current master database version box appears above Existing/old project database because it is the base file for New Project and Upgrade Project. The Existing/old project database box is used by Upgrade Project and Existing Project. Upgrade Project is the only mode that needs both boxes.

Main Tab Order

Project Pickle is arranged left to right as: Project, Periods, AppColumns, Upgrade Mapping, Plots, Get Field Order, Reports, CFIDEER Queries, Plot Assignments, Tree Assignments, Processing, Build, and AI Help.

Reports appears after Get Field Order so users can finish AppColumns and query-field order setup before arranging the matching report columns and fixed-width headers.

Processing appears after the assignment tabs because it is an optional Upgrade Project step for carrying completed Processing setup tables and optional Plot, Tree, and Regen calculation and summary tables into the upgraded output database.

Use the Next > button in the bottom-right status area to move through the tabs. Project Pickle checks required entries on the current tab before moving forward.

When Next > is used on the Plots tab, Project Pickle moves directly to Get Field Order. From there, the workflow continues to Reports, CFIDEER Queries, assignments, optional upgrade-only Processing, and Build.

Periods is read-only and skipped in Existing Project mode.

Upgrade Mapping, Tree Assignments, and Processing are greyed out and skipped unless Upgrade Project mode is selected.

The manual Reports designer is available in all three project modes. The completed report-table copy option is available only in Upgrade Project mode.

Copy AppColumns from existing database can be used in Upgrade Project or Existing Project mode. It cannot be used in New Project mode because New Project has no Existing/old project database.

The Processing tab uses a pink accent and the Build tab uses a red accent so the two final workflow areas are easier to locate. The colors do not change which options are available.

Before You Build

Before clicking Build, make sure you know these items:

  • Which mode you need.
  • Which database box or boxes the selected mode requires.
  • Where the saved project database should be saved.
  • The project name for New Project or Upgrade Project work. Existing Project mode keeps the project name already stored in the selected database.
  • The region, agency, and reservation name for New Project work, or whether Project Pickle should keep those values from an existing database.
  • The measurement period or periods that should exist in the output.
  • Whether plots are ready now or will be added later.
  • Whether AppColumns/custom fields are ready now or will be added later.
  • Whether assignments are ready now or should be skipped until plot data exists.
  • Whether CFIDEER queries should be rebuilt, copied from the old database during an upgrade, or left unchanged.
  • In Upgrade Project mode, whether either Processing table group should be copied from the old database.

You can skip optional work. If plots, assignments, AppColumns, or CFIDEER queries are not ready, leave them blank or off. You can run Project Pickle again later in Existing Project mode using the current project database as the Existing/old project database and a new output copy.

Upgrade-only copy choices must be decided during the upgrade. Copying old _prjCFIDEER_ queries and importing Processing tables are disabled in Existing Project mode. Leaving them off during an upgrade means the upgraded output keeps the corresponding definitions or rows from the current master database version.

Project Tab

The Project tab is the main setup screen. It tells Project Pickle what database to read, where to save the output, and which region/agency/reservation/project information to use.

What To Fill Out

  • Mode: choose New Project, Upgrade Project, or Existing Project.
  • Current master database version: used in New Project and Upgrade Project mode. New Project copies this database as the starting point. Upgrade Project copies this database, then imports data from the old project database. This row is greyed out in Existing Project mode.
  • Existing/old project database: used in Upgrade Project and Existing Project mode. In Upgrade Project, choose the old database you are moving data out of. In Existing Project, choose the current-format project database you want to update. This row is greyed out in New Project mode.
  • Save Project Location: choose where the finished database copy should be saved. Before a real project name is entered, the Browse window builds the suggested name from the current master database version for New/Upgrade mode or from the selected project database for Existing mode, but leaves the version as a fill-in placeholder, such as [TribeName] CFI [Date 20260520] vX.XX.XX.mdb.
  • Project name: enter the project name for New Project or Upgrade Project mode. Project Pickle also uses this name in the Access Application Title, which is the name shown in the top bar when the saved project database opens in Microsoft Access. In Existing Project mode, this box is read-only and Project Pickle keeps the project name and Access Application Title already stored in the selected database.
  • Measurement cycle interval (years): enter the planned number of years between full CFI measurements for New Project or Upgrade Project mode. This is not the number of periods. In Existing Project mode, this box is read-only and Project Pickle keeps the existing value.
  • Region, Agency, Reservation name: load and select the correct values from the chosen database for New Project mode. In Existing Project mode, these boxes are read-only because the selected project database should already contain the correct location.

Loading Region, Agency, And Reservation Lists

  1. Choose the database required by the selected mode. New Project uses Current master database version. Upgrade and Existing use Existing/old project database for these lists.
  2. Click Load lists.
  3. For New Project mode, select the correct Region.
  4. For New Project mode, select the correct Agency.
  5. For New Project mode, select the correct Reservation name.
  6. For Existing Project mode, Project Pickle displays the existing Project name, cycle interval, Region, Agency, and Reservation as read-only values.

The lists are full lists from the selected database. They do not have to filter each other. The selected reservation supplies the reservation and agency links Project Pickle needs during setup.

Reference Rows Are Cleaned Automatically

Project Pickle automatically keeps only the selected Region, Agency, and Reservation rows in the saved project database. This is the normal setup behavior and helps keep the project database focused on the selected location.

The Keep Region, Agency, and Reservation from Existing Database box controls whether those location values are read from the selected existing database instead of selected manually. In New Project mode, the box is off and locked because a brand-new project needs manual Region, Agency, and Reservation selections. In Upgrade Project mode, the box is on by default so the upgraded output keeps the old project's location. In Existing Project mode, the box is on and locked because the selected project database should already contain the correct location. Existing Project mode also preserves the existing project name and cycle interval instead of rewriting them from the Project tab.

Periods Tab

The Periods tab creates or reviews project measurement period rows for New Project and Upgrade Project work only.

For A New Project

  • Most new projects start with one period.
  • Enter the project information in row 1 with PeriodNumber 1.
  • Check IsCurrent for that row.
  • Dates can be blank if they are not known yet.

For Upgrade Or Remeasurement Work

  • Click Load periods from database to load periods from the old project database.
  • Verify the period information and number of periods.
  • A project with five previous measurements and a new sixth measurement should have periods 1 through 6.
  • Add the new period for the remeasurement and check IsCurrent on that new period.
  • The Load periods from database button is enabled only in Upgrade Project mode. It reloads period rows from the old project database and is disabled for New Project mode so sample periods from a master database are not imported.
  • Only one period can be current, so Project Pickle unchecks the others automatically.
  • Project Pickle preserves hidden period IDs where possible so old measurement rows still link to the correct period.

In Existing Project mode, the Periods tab is read-only/greyed out because the selected project database should already contain the correct period structure. Do not use Existing Project mode to add a new measurement period.

Standard Existing Project workflow: use Existing Project mode only for adding new field data items or additional plots, then continue through Get Field Order, CFIDEER Queries, Plot Assignments, and Build.

When you add a new period row by hand, Project Pickle checks that new row as current automatically. If that is not the correct current period, check the correct row and Project Pickle will clear the other selection.

After Project Pickle writes the ProjectMeasurementPeriods rows, it updates every Analysis* table that has a ProjectPeriodID field so it uses the current period's newly created ProjectPeriodID.

ApplicablePeriod

ApplicablePeriod is used when Project Pickle updates ValidationRules.ApplicablePeriods. Use the period value your project expects for validation rules, often the measurement year or period identifier.

Is Current

Is Current marks the period that is currently being set up or measured. Use it for the active measurement period.

Upgrade Mapping Tab

The Upgrade Mapping tab matches older database field names to the new standard field names and adds older fields that are not collected in the current measurement period into the custom tables, preserving them as part of the project history.

The Upgrade Mapping grid has its own bottom scrollbar because old table names, old field names, target table choices, and notes can be wider than the window.

Use Upgrade Mapping for older history fields. Only add AppColumns for fields that will be collected in the current measurement. Older fields that still need to exist in the upgraded database should be created through Upgrade Mapping instead.

If an older field has a different name and needs to land in a custom table, add that new custom AppColumns field first. Then return to Upgrade Mapping and select it in the New table or table.field column.

Upgrade Mapping only runs in Upgrade Project mode. If the run log says Mode = Existing Project, Project Pickle did not run the upgrade import or field-moving step. Go back to the Project tab, choose Upgrade Project, choose the old project database in Existing/old project database, choose the current master database version, then build again.

During upgrade import, Project Pickle tries to preserve full plot, tree, and regen rows. If Access rejects one field value, Project Pickle tries to keep the row and writes the skipped field to the run log so the user can review it after the build.

When a same-name field moved between a standard measurement table and its matching custom measurement table, Project Pickle normally uses fast bulk Access queries to copy those values. The run log says Upgrade moved-field bulk mapping when that faster route is used. If Access rejects the bulk query because an older database has an unusual table shape or field type, Project Pickle writes that to the log and automatically uses the slower row-by-row fallback.

Use Check data types before building an upgrade. Project Pickle checks old source values against the current master database version, active custom fields, automatic moved-field targets, and the choices in this tab. It warns about datatype changes, such as old text values being sent to a new integer field. Project Pickle now stops the build for these mismatches even when the current values look convertible, because fields such as old text HabitatType should be crosswalked into the new integer code before they are written to the standard new field.

If the crosswalk is ready, convert the old values first and then run Project Pickle. If the crosswalk is not ready, change the Upgrade Mapping target to a Text/Memo custom destination, such as choosing PlotCustomMeasurements by table name for old Plots.HabitatType. That preserves the old text so it can be crosswalked later without forcing it into the new integer field.

Project Pickle also checks for measurement rows whose parent inventory rows are missing. For example, an old database may have a row in TreeMeasurements but no matching row in Trees. When enough information exists, Project Pickle rebuilds missing Plots and Trees rows, marks UpdUser=Pickle, and writes the repaired IDs to the run log. Project Pickle also creates missing matching custom measurement rows when needed.

When To Use It

  • Use it in Upgrade mode.
  • Use it when the run log says old fields were skipped or need review.
  • Use it when a field was renamed between database versions.
  • Use it when a field moved between related plot, tree, or regen tables and Project Pickle cannot match it by the same name.

How To Map Fields

  1. Choose the old project database on the Project tab.
  2. Choose the current master database version on the Project tab.
  3. If the new target field already exists under a different name, you can map directly to that exact field.
  4. Return to Upgrade Mapping.
  5. Click Load unmapped fields.
  6. Use Load saved mapping file only if you already have a saved Project Pickle mapping JSON file from a similar upgrade.
  7. Click Check data types to find old values that will not fit the selected new target type.
  8. For each old field, choose either a target table or an exact Table.Field target.
  9. The most common table targets appear near the top of the list: PlotCustomMeasurements, TreeCustomMeasurements, Plots, Trees, PlotMeasurements, and TreeMeasurements.
  10. Project Pickle can map within the same data family, such as plot measurement to plot custom measurement, plot custom measurement to plots, tree custom measurement to tree measurement, or tree custom measurement to trees.
  11. Project Pickle skips fields named NeedsReview and common standard fields that should not be manually mapped. Those NeedsReview fields come from the old database table layout; Project Pickle does not create them.
  12. Project Pickle preselects likely targets when it loads the list. Old PlotCustomMeasurements fields are suggested for new PlotCustomMeasurements, and old TreeCustomMeasurements fields are suggested for new TreeCustomMeasurements.
  13. Old TreeRemarks is suggested for TreeMeasurements.RemarksTree.
  14. Old PlotRemarks is suggested for PlotMeasurements.RemarksPlot.
  15. Old regen Remarks is suggested for RegenMeasurements.RemarksRegen.
  16. Old Plots.HabitatType is suggested for PlotMeasurements.HabitatType.
  17. Old FieldOverCover is suggested for PlotMeasurements.CoverType.
  18. Old FieldOverDensity is suggested for PlotMeasurements.DensityClass.
  19. Old FieldOverSize is suggested for PlotMeasurements.SizeClass.
  20. Old ForestClassification is suggested for Plots.FLCCommercial.
  21. Old unmapped TreeMeasurements fields are suggested for TreeCustomMeasurements.
  22. Old unmapped Plots and PlotMeasurements fields are suggested for PlotCustomMeasurements.
  23. Choose a table, such as PlotCustomMeasurements, TreeCustomMeasurements, Plots, or Trees, when Project Pickle should add the old field to that table using the old field name.
  24. When source and target tables use different row keys, Project Pickle uses related IDs and keys such as MeasurementID, PlotMeasKey, TreeMeasKey, RegenMeasKey, PlotID, PlotKey, TreeID, and TreeKey to find the matching rows.
  25. Choose Table.Field, such as TreeCustomMeasurements.ExampleField, when the old field should be renamed during import.
  26. Choose Skip for fields that should not carry forward. Skip means Project Pickle will not create that old field in the output custom tables and will not copy data for that field. Use it when the old field is empty, has no useful data, or should not be preserved in the upgraded database.
  27. Save the mapping to JSON only if you want to reuse it for similar upgrades.

NeedsReview fields are skipped automatically. They come from some older database table layouts and should not be carried forward into the current standard structure.

Upgrade Mapping copies old values into the selected output table and can add the physical field when needed. It is for moving data between related output tables, not for turning fields on in AppColumns. If a mapped field should be active for data entry, reports, or CFIDEER queries, add or activate that field on the AppColumns tab separately.

Save mapping is optional. It saves the current grid as a reusable JSON file. The Build button uses the mappings currently shown in the grid even if you do not save. If you close Project Pickle before building, unsaved mapping choices are lost.

Load saved mapping file opens a file picker because it is only for reusing a mapping JSON file that was created earlier with Save mapping. It is not used to choose the old project database, the current master database version, or a plot spreadsheet. For normal upgrades, start with Load unmapped fields.

Plots Tab

The Plots tab is used to add new plots and import plot data from a previous project.

If you do not have plot data yet, leave the Plots tab blank. You can build the database now and come back later in Existing Project mode when plot data is available.

New Project

Add plots from a spreadsheet. Project Pickle sets up the Plots table and creates initial PlotMeasurements and PlotCustomMeasurements records.

Upgrade Projects

Project Pickle imports plots from the previous database into the Plots, PlotMeasurements, and PlotCustomMeasurements tables. It also creates new records for the current measurement in the PlotMeasurements and PlotCustomMeasurements tables.

Existing Project Optional Step

In Existing Project mode, the Plots tab is optional. If no new plots need to be added, leave Add New Plots blank and skip this step. The selected current-format project database already contains all existing plots.

If the selected database was already built through Upgrade Project mode, do not re-import the old project data. Use Add New Plots only when the tribe is adding new plot numbers that are not already in the database.

Adding New Plots In Upgrade Or Existing Projects

Add plots from a spreadsheet using plot number and any additional information provided. Project Pickle creates blank records for the existing period rows already in the selected project setup so the new plots line up with the project period structure.

You can upload a spreadsheet that contains only the new plots. You do not need to include plots that are already in the database. You may also upload a mixed spreadsheet with existing and new plots; Project Pickle skips plot numbers that already exist and adds only the new plot numbers.

The Import Plot, Tree, Regen, Assignment Data from the previous project checkbox is Upgrade-only. It is checked automatically in Upgrade Project mode and disabled in Existing Project mode on purpose.

Importing Plots From A Spreadsheet

The file box on this tab is called Add New Plots. Use it when you have a spreadsheet of new plot numbers to add to the project.

  1. Select the plot spreadsheet or CSV file.
  2. Click Load columns.
  3. For each Project Pickle field, choose the spreadsheet column that contains that value.
  4. PlotNumber is required.
  5. Optional mapped fields include PlotLabel, PlotTypeID, UTMNorthingCoordinate, UTMEastingCoordinate, UTMZone, and FLCCommercial.

The spreadsheet does not have to use the exact Project Pickle field names. The mapping step tells Project Pickle which spreadsheet column goes with each database field.

The mapping grid has its own bottom scrollbar. Use it when the app is not maximized or when long spreadsheet column names push fields off screen.

If the build says a row is missing PlotNumber, click Load columns and confirm PlotNumber is mapped to the spreadsheet column with plot numbers. Project Pickle skips fully blank spreadsheet rows, but a row with other mapped data still needs a plot number.

Adding New Plots To An Existing Project

If an existing project already has 200 plots and the tribe adds 50 more, upload a spreadsheet with just those 50 new plots. Project Pickle keeps the 200 existing plots, adds the 50 new plot numbers, and creates blank PlotMeasurements and matching PlotCustomMeasurements rows for the period rows already in the selected project database.

Blank Plot Measurement Rows For A New Period

When New Project or Upgrade Project mode creates a new period, Project Pickle creates blank PlotMeasurements rows for missing plot/period combinations. It fills the required IDs and keys, ProjectPeriodID, PeriodNumber, and UpdUser=Pickle. Other measurement fields stay blank. Project Pickle also creates a matching PlotCustomMeasurements row with the same MeasurementID. Existing Project mode does not add new periods; it uses the period rows already in the selected project database.

AppColumns Tab

The AppColumns tab controls which fields are active in the logger/data-entry setup, which fields show in reports, and which fields show in CFIDEER queries.

Use AppColumns for the current measurement. Only add or activate AppColumns for fields that users will collect now. Older fields that should remain in the database for project history should be preserved through the Upgrade Mapping tab instead.

If an older field needs a different new custom field name, add that custom field here first so it can be selected later in Upgrade Mapping under New table or table.field.

CFIDEER Get/Update queries are built from fields that are both Active and QueryVisible. Fields created only through Upgrade Mapping can stay in the physical custom tables to preserve older project history, but they will not be added to CFIDEER queries unless they are also active AppColumns for the current measurement.

Before building with CFIDEER query refresh turned on, Project Pickle shows a confirmation listing the custom fields that are currently Active + QueryVisible. Those listed fields will be added to the _prjCFIDEER Get/Update queries. If an older history field appears in that popup, cancel the build and uncheck Active or QueryVisible before running. Fields that are inactive or not query-visible remain in the custom tables only for history.

After fields are added, Project Pickle orders the physical custom measurement tables so required key fields appear first, active AppColumns appear next, and inactive Upgrade Mapping history fields appear last. This keeps newly activated fields ahead of older inactive fields in PlotCustomMeasurements, TreeCustomMeasurements, and RegenCustomMeasurements.

This tab has two main boxes:

  • Load Standard AppColumn: the standard/current AppColumns list from the current master or current-format project database.
  • New custom AppColumns to add: new project-specific custom fields you want Project Pickle to create.

Before Project Pickle changes AppColumns, it creates a fresh AppColumnsArchive table inside the saved project database. This is a before-build snapshot from the base database copied for the current run. If the archive already exists in the output copy, Project Pickle replaces it with a new snapshot for the current run.

Copy AppColumns From Existing Database

The Copy AppColumns from existing database checkbox is for a project whose AppColumns setup is already complete in another project database and should be carried forward without rebuilding every row manually.

This option can be used in Upgrade Project mode and Existing Project mode. It cannot be used in New Project mode because there is no Existing/old project database to copy from.

This is a complete replacement, not a merge. During Build, Project Pickle replaces all rows in the output copy's AppColumns and AppColumnCodes tables with rows from the selected Existing/old project database.

When the checkbox is selected, manual AppColumns loading and editing are locked for that build. This includes loading the standard list, changing loaded settings, editing code lists, adding new custom rows, and using Copy selected format.

Project Pickle still creates AppColumnsArchive before the replacement. The archive is a snapshot of the AppColumns table originally present in the output copy's base database. In Upgrade Project mode, this is normally the current master database version's AppColumns setup, not a second copy of the old project's table.

During Build, Project Pickle:

  1. Creates or refreshes AppColumnsArchive.
  2. Deletes existing output AppColumnCodes rows.
  3. Deletes existing output AppColumns rows.
  4. Copies all compatible AppColumns rows from the Existing/old project database.
  5. Copies all compatible AppColumnCodes rows from the Existing/old project database.
  6. Checks copied custom fields against PlotCustomMeasurements, TreeCustomMeasurements, and RegenCustomMeasurements.
  7. Adds a missing physical custom-table field when a copied custom AppColumns row requires it.
  8. Writes source and output row counts to the run log.

The replacement is performed in a transaction so a failed copy does not leave only part of the AppColumns setup in the output.

Copying AppColumns does not by itself copy Plot, Tree, or Regen measurement records. Measurement data is handled by the normal inventory import and upgrade workflow.

When completed reports are also being copied and those reports use custom fields, select Copy AppColumns from existing database too. ReportColumns rows reference AppColumns IDs, so matching AppColumns must exist in the output.

Archive Old AppColumns And AppColumnCodes During An Upgrade

The checkbox Archive old AppColumns and AppColumnCodes as zAppColumns and zAppColumnsCodes is available only in Upgrade Project mode. It is checked by default when Project Pickle starts.

This archive option is independent from Copy AppColumns from existing database. A user can:

  • Archive the old setup without replacing the working AppColumns.
  • Archive the old setup and also copy it into the working AppColumns.
  • Uncheck the archive option and continue the upgrade without creating or refreshing the two z tables.
Table What It Contains Purpose
AppColumnsArchive The AppColumns setup originally present in the base database copied to the output. Preserves the before-build AppColumns setup from the current master database version or current-format project database.
zAppColumns Every column and row from AppColumns in the selected Existing/old project database. Preserves the old project's AppColumns setup for later review.
zAppColumnsCodes Every column and row from the old database's AppColumnCodes table. Preserves the old project's code-list setup for later review.

When the archive option is checked, Build:

  1. Verifies that the selected Existing/old project database contains AppColumns and an AppColumnCodes table.
  2. Reads the source row counts.
  3. Removes prior zAppColumnsCodes and zAppColumns tables from the output copy when they already exist.
  4. Creates fresh zAppColumns and zAppColumnsCodes tables in the output database.
  5. Compares the source and archived row counts.
  6. Writes both archived row counts to the detailed Build log.
  7. Removes both archive tables if an error occurs after only part of the two-table archive was created.

The z tables are reference snapshots. Selecting this archive option does not activate fields, change report visibility, change query visibility, replace working AppColumns, or replace working AppColumnCodes.

The archive preserves source fields and rows. It is not intended to recreate working-table indexes, relationships, primary-key behavior, or AutoNumber behavior.

If the archive option is selected but the old database does not contain the required AppColumns or AppColumnCodes source table, Project Pickle stops the Build rather than creating an incomplete one-table archive. Choose a compatible source database or uncheck the archive option.

Do not confuse the two copy commands:

  • Copy selected format copies table placement, data type, and category from one selected row into one new custom row.
  • Copy AppColumns from existing database replaces the complete AppColumns and AppColumnCodes tables during Build.

Reviewing Existing AppColumns

Load Standard AppColumn fills the top review grid from the current master/current-format AppColumns table. There is no separate Existing AppColumn list. Existing/project rows show in the same top grid only after they already exist in the selected database, usually because custom AppColumns were added during a previous Upgrade Project or Existing Project run. New custom rows stay in the lower box until you build; after that, they become existing AppColumns in future runs.

In the loaded review box, App table and StandardCode are read-only. App table shows where the standard AppColumns row belongs, and StandardCode shows whether a loaded Code row uses a standard code/rule list. Use the lower custom box when a new field needs a selectable App table.

  1. Choose the database required by the selected mode on the Project tab.
  2. In Upgrade mode, also choose the current master database version.
  3. Go to the AppColumns tab.
  4. Click Load Standard AppColumn.
  5. Use the search box to find fields by field name.
  6. Use the field filters to show all fields, active fields only, inactive fields only, plot fields, tree fields, or regen fields.
  7. Uncheck Active for loaded standard fields that do not apply to the project.

Fields Project Pickle Hides From Review

Project Pickle hides fields that are always required, standard, calculated, or not useful to turn on and off. This includes standard regen fields such as RemarksRegen and StemCount. This keeps the review list focused on fields a user may actually need to decide about.

Adding A New Custom Field

  1. Use one row per custom field that will be collected in the current measurement and is not already in the standard list above.
  2. Do not add old fields that will not be collected from an older database. Use Upgrade Mapping to create those legacy fields in the appropriate custom table.
  3. If an older field needs a different new custom field name, add that custom field here first so it can be selected later in Upgrade Mapping under New table or table.field.
  4. Choose the app/custom table, such as plot, tree, or regen custom measurements.
  5. Enter the field name using PascalCase, such as CountyCode. Do not use spaces, underscores, or punctuation.
  6. Choose the data type.
  7. Choose the category.
  8. Make sure Active is checked for fields that should be collected.
  9. If the category is Code, enter all allowed codes in the Codes cell.
  10. Leave Active, ReportVisible, and QueryVisible checked for a typical custom field. If you check Active after it was off, Project Pickle also checks ReportVisible and QueryVisible. If you uncheck Active, Project Pickle also unchecks ReportVisible and QueryVisible.
  11. To remove a new custom row before building, select the row and press Delete. Project Pickle shows a warning before removing it from the setup list. This does not remove fields from a database that has already been built.

Copy Selected Format

If an existing field has the same kind of setup you need, select it in the loaded review box and click Copy selected format. Project Pickle copies the table placement, data type, and category into a new custom row. It does not copy the old field name or code list.

Checkbox Meanings

Active

Field is on and collected for the current project. Inactive fields stay out of CFIDEER queries even if the physical field exists in a custom table. Checking Active also checks ReportVisible and QueryVisible; unchecking Active also unchecks them.

ReportVisible

Makes an active field available to reports. The automatic report updater can include it, and the Reports tab's Add ReportVisible fields button can add it to the matching Plot, Tree, or Regen manual layout.

Checking ReportVisible does not immediately add the field to an already saved manual report layout. Open Reports, choose the matching layout, click Add ReportVisible fields, arrange the field, update the formats and header text, then save the layout.

QueryVisible

Project Pickle can include the active field when rebuilding CFIDEER get/update queries. Active must also be checked.

StandardCode

Read-only. The loaded existing row uses a standard code/rule list. This appears only in the loaded review area.

Entering Multiple Codes Correctly

For one field, put all allowed codes in that field's single Codes cell. Use code=label pairs separated by semicolons.

Example with three codes:

1=test1; 2=test2; 3=test3

Example with a longer code list:

0=None; 1=Trace; 2=Light; 3=Moderate; 4=Heavy; 5=Severe; 6=Dead; 7=Broken; 8=Missing; 9=Unknown; 10=Other; 11=Not applicable

You can also paste one code per line. Project Pickle should create one AppColumnCodes row for each code.

When Project Pickle loads standard AppColumns, code lists are shown in numeric order, such as 0, 1, 2 ... 10 ... 100, instead of text order.

The tree species code list for AppColumns is still updated manually in the Access database after Project Pickle finishes running. Project Pickle can build the AppColumns field setup, but the final tree species code list maintenance is still a manual database step.

Adding And Ordering Report Fields

  • Tree fields before TreeMeasurements.RemarksTree
  • Regen fields before RegenMeasurements.RemarksRegen

Fixed-Width ReportHeaders Editor

The lower editor contains the exact text Project Pickle writes to ReportHeaders.Header.

Spaces are data. Project Pickle preserves repeated spaces and does not trim or collapse them.

Use the character ruler and live line-length message to align labels, vertical bars, and dash lines. Every nonblank line in one layout must be the same character length.

  • Use spaces rather than tabs.
  • Use printable ASCII characters.
  • Do not place an embedded line break inside one line.
  • Keep every line at or below 255 characters.
  • Keep every nonblank line the same length.
  • Keep custom fields before the required remarks field.

Blank lines are allowed and are written as separate ReportHeaders records.

Save Current Layout

Save current layout validates and stores the selected layout in the app's memory. It does not immediately change a database.

Save Plot, Tree, and Regen separately after editing each layout. The saved layouts are written only when Build runs with report updating enabled.

Manual report layouts are not saved to a separate file. Closing Project Pickle before Build loses the in-memory report edits.

Automatic Space Before The Remarks Field

Project Pickle automatically protects the boundary between the final custom report field and the required Remarks field.

The rule applies when the field immediately before Remarks belongs to the matching custom measurement table:

  • PlotCustomMeasurements immediately before RemarksPlot
  • TreeCustomMeasurements immediately before RemarksTree
  • RegenCustomMeasurements immediately before RemarksRegen

Users continue to see and edit normal FieldFormat values in the Reports grid. During preview, validation, Save Current Layout, and Build, Project Pickle converts the final eligible custom field to a stored format that guarantees one trailing blank.

Normal Editable Format Stored/Printed Format Purpose
{0,1} {0,2:0\ } Keeps the original one-character value position and adds one guaranteed blank before Remarks.
{0,4:g0} {0,5:0\ } Keeps the original four-character numeric area and adds one guaranteed blank before Remarks.

Users do not need to type the slash format. Continue entering normal widths with Edit width / tally. Project Pickle applies the special stored format automatically only to the final eligible custom field before Remarks.

The special format is converted back to a normal editable format when the layout is displayed again, so repeatedly loading and saving a layout does not keep increasing the width.

Specialized text, date, and other non-whole-number legacy formats are left unchanged. A standard field immediately before Remarks is also left unchanged.

RemarksTree Format Protection

TreeMeasurements.RemarksTree must remain:

TEXT,40,5

Project Pickle restores that value in both the visible Reports grid and the current in-memory Tree layout. This protection also runs after Add ReportVisible fields, so adding a custom Tree field does not convert RemarksTree to a numeric format.

The automatic separator adds one printed character to the final custom field's stored width. After adding ReportVisible fields, open the synthetic preview and use Validate all layouts. Adjust every nonblank ReportHeaders line so it matches the final printed width.

Validate All Layouts

Validate all layouts checks loaded manual layouts for missing columns, duplicate fields, missing FieldFormat values, missing remarks fields, custom fields after remarks, missing header text, tabs, embedded line breaks, unsupported characters, lines longer than 255 characters, and unequal nonblank line lengths.

How Build Chooses The Report Method

  1. If completed report-table copy is selected, Project Pickle copies the complete report tables and skips both manual and automatic report generation.
  2. Otherwise, when Update reports is checked and manual layouts have been saved, Project Pickle writes those layouts exactly.
  3. Otherwise, when Update reports is checked but no manual layouts were saved, Project Pickle uses the legacy automatic report updater.
  4. When Update reports is unchecked, Project Pickle keeps the report setup inherited from the base database copy.

Copy Completed Reports During An Upgrade

The checkbox Copy completed ReportHeaders and ReportColumns from existing database appears only in Upgrade Project mode.

Use it when the old project database already contains fully completed and tested report tables that should be carried into the upgraded output.

When it is checked:

  • Manual Reports-tab controls are disabled.
  • The Build report option is forced on and locked.
  • Manual layouts are ignored.
  • The automatic report generator is skipped.
  • All rows in ReportHeaders and ReportColumns are replaced.
  • Fixed-width spaces and blank Regen group values are preserved.
  • The replacement is performed in one transaction.

The copy includes every report section and group found in the old database, not only the three layouts supported by the manual designer.

Project Pickle preserves a ReportColumns ColumnID when the same AppColumns ID exists in the output. When the GUID changed, it tries to match the field by AppTables table name and AppColumns column name.

If a copied report references an AppColumns field that is missing from the output, Build stops before replacing the report tables. Select Copy AppColumns from existing database too, or create the matching AppColumns first.

CFIDEER Queries Tab

Summary

The CFIDEER Queries tab controls whether Project Pickle rebuilds the saved Get and Update queries used by CFIDEER/data-entry workflows.

Normal Query Rebuild

For a normal rebuild, Project Pickle reads the current measurement tables, the active and QueryVisible AppColumns, and the order loaded on the Get Field Order tab. It then generates the _prjCFIDEER_ Get/Update SQL and replaces the working query definitions in the output database.

Build/refresh _prjCFIDEER_ get and update queries is normally checked by default. Leave it checked when the queries should match the current AppColumns and table setup.

Before rebuilding, click Load fields from AppColumns setup on the Get Field Order tab. Rebuilding without loading the field-order lists can leave the query builder without the field-order information it needs.

Copy Old Query Definitions During An Upgrade

When Copy existing _prjCFIDEER_ queries from old database is selected on the Get Field Order tab, the normal query-rebuild checkbox is automatically unchecked and locked. Project Pickle copies the saved old query definitions and skips query generation.

The two choices cannot be used together:

  • Copy old queries: use the old database's saved query names and SQL.
  • Rebuild queries: use current AppColumns, table structure, and Get Field Order.

If both options are off, Project Pickle does not change the query definitions inherited from the base database copy. In Upgrade Project mode, that base is the current master database version. In Existing Project mode, that base is the selected current-format project database.

Important

Both query copy and query rebuild update only the output copy. Query rebuilding overwrites the working query definitions without creating separate query backups. Query copy replaces matching _prjCFIDEER_ definitions with definitions from the old database. Always review the resulting queries before production data entry.

Updating Query Fields Later

To add or remove fields from the queries later, run Project Pickle in Existing Project mode, choose a new output copy, update Active and QueryVisible on the AppColumns tab, load Get Field Order, and leave the normal query-rebuild option checked.

For _prjCFIDEER_GetProcessPlotList, _prjCFIDEER_GetProcessTreeList, and _prjCFIDEER_GetProcessRegenList, Project Pickle removes prior explicit custom-table select items from the saved process-list query, then adds back only fields that are active and query-visible in the AppColumns setup. Older fields created through Upgrade Mapping stay in the custom tables for history, but they are not carried into the process-list queries unless they are collected in the current measurement.

For _prjCFIDEER_GetPlotMeasurementsForPeriod, _prjCFIDEER_GetTreeMeasurementsForPeriodByKey, and _prjCFIDEER_GetRegenMeasurements, the same rule applies: only fields that are both Active and QueryVisible are included. Upgrade Mapping history fields should usually stay inactive and/or not query-visible.

For _prjCFIDEER_GetPlotMeasurementsForPeriod, Project Pickle starts the parameter declarations with PlotID Guid and ProjectPeriodID Guid, followed by SELECT. Its FROM block uses PlotMeasurements LEFT JOIN PlotCustomMeasurements ON PlotMeasurements.[MeasurementID] = PlotCustomMeasurements.[MeasurementID], and its criteria check PlotMeasurements.PlotID = [PlotID] before PlotMeasurements.ProjectPeriodID = [ProjectPeriodID]. PlotMeasurements.PlotID and PlotMeasurements.ProjectPeriodID are criteria-only fields, so their Access query design Show boxes are unchecked by default. The select list does not include CalcSiteIndex, GMP, Created, Updated, UpdUser, or RetrievalID, and RemarksPlot is always the last selected field before the criteria-only PlotID/ProjectPeriodID fields.

For _prjCFIDEER_GetTreeMeasurementsForPeriodByKey, Project Pickle starts the parameter declarations with TreeKey Text (255) and PeriodID Guid, followed by SELECT. Its FROM block uses TreeMeasurements LEFT JOIN TreeCustomMeasurements ON TreeMeasurements.MeasurementID = TreeCustomMeasurements.MeasurementID, and its criteria check TreeMeasurements.TreeKey = [Treekey] before TreeMeasurements.ProjectPeriodID = [PeriodID]. TreeMeasurements.TreeKey and TreeMeasurements.ProjectPeriodID are criteria-only fields, so their Access query design Show boxes are unchecked by default. The select list does not include Created, Updated, UpdUser, or RetrievalID, and RemarksTree is always the last selected field before the criteria-only TreeKey/ProjectPeriodID fields.

For _prjCFIDEER_GetRegenMeasurements, Project Pickle starts the parameter declarations with RegenID Guid, followed by SELECT. It intentionally leaves out RegenMeasurements.PlotNumber, RegenMeasurements.PlotID, RegenMeasurements.ProjectPeriodID, IsDeleted, Created, Updated, UpdUser, and RetrievalID, and filters by RegenMeasurements.MeasurementID = [RegenID]. RegenMeasurements.MeasurementID is criteria-only, so its Access query design Show box is unchecked by default. RemarksRegen is always the last selected field before the criteria-only MeasurementID field.

For _prjCFIDEER_UpdatePlotMeasurement, Project Pickle starts the update fields with PlotMeasurements.PlotMeasKey, PlotMeasurements.PlotKey, and PlotMeasurements.PeriodNumber. Other in-house apps depend on those fields staying at the start of that update query. Project Pickle does not include CalcSiteIndex or GMP in this update query so the update field count stays aligned with the Get Plot measurement query.

For _prjCFIDEER_UpdatePlotMeasurement, Project Pickle starts the parameter declarations with PlotID Guid, PeriodID Guid, PlotMeasKey Text (255), and PlotKey Text (255).

For _prjCFIDEER_UpdatePlotMeasurement, Project Pickle starts the update SQL body with UPDATE PlotMeasurements, then INNER JOIN PlotCustomMeasurements ON PlotMeasurements.MeasurementID = PlotCustomMeasurements.MeasurementID, then a multiline SET block beginning with PlotMeasurements.PlotMeasKey, PlotMeasurements.PlotKey, and PlotMeasurements.PeriodNumber.

For _prjCFIDEER_UpdatePlotMeasurement, Project Pickle also ends the update grid with PlotMeasurements.Created, PlotMeasurements.Updated, PlotMeasurements.UpdUser, PlotMeasurements.RetrievalID, PlotCustomMeasurements.PlotMeasKey, then the criteria fields PlotMeasurements.PlotID and PlotMeasurements.ProjectPeriodID.

For _prjCFIDEER_UpdateTreeMeasurement, Project Pickle starts the update fields with TreeMeasurements.TreeMeasKey, TreeMeasurements.TreeKey, and TreeMeasurements.PeriodNumber.

For _prjCFIDEER_UpdateTreeMeasurement, Project Pickle starts the parameter declarations with TreeID Guid, PeriodID Guid, TreeMeasKey Text (255), TreeKey Text (255), and PeriodNumber Short.

For _prjCFIDEER_UpdateTreeMeasurement, Project Pickle starts the update SQL body with UPDATE TreeMeasurements, then INNER JOIN TreeCustomMeasurements ON TreeMeasurements.MeasurementID = TreeCustomMeasurements.MeasurementID, then a multiline SET block beginning with TreeMeasurements.TreeMeasKey, TreeMeasurements.TreeKey, and TreeMeasurements.PeriodNumber.

For _prjCFIDEER_UpdateTreeMeasurement, Project Pickle does not include Created in the update grid or SQL. It ends the update grid with TreeMeasurements.Updated, TreeMeasurements.UpdUser, TreeMeasurements.RetrievalID, TreeCustomMeasurements.TreeMeasKey, then the criteria fields TreeMeasurements.TreeID and TreeMeasurements.ProjectPeriodID. Updated updates from [UpdatedDate], and ProjectPeriodID uses the [PeriodID] criteria parameter.

For _prjCFIDEER_UpdateRegenMeasurement, Project Pickle starts the update fields with RegenMeasurements.RegenMeasKey, RegenMeasurements.PlotID, RegenMeasurements.PlotKey, and RegenMeasurements.ProjectPeriodID. The ProjectPeriodID field updates from the [PeriodID] parameter. Project Pickle does not include IsDeleted in this update query because including it can prevent regen data from opening or saving back correctly.

For _prjCFIDEER_UpdateRegenMeasurement, Project Pickle starts the parameter declarations with RegenMeasurementID Guid, RegenMeasKey Text (255), and PlotID Guid.

For _prjCFIDEER_UpdateRegenMeasurement, Project Pickle starts the update SQL body with UPDATE RegenMeasurements, then INNER JOIN RegenCustomMeasurements ON RegenMeasurements.MeasurementID = RegenCustomMeasurements.MeasurementID, then a multiline SET block beginning with RegenMeasurements.RegenMeasKey, RegenMeasurements.PlotID, RegenMeasurements.PlotKey, and RegenMeasurements.ProjectPeriodID.

For _prjCFIDEER_UpdateRegenMeasurement, Project Pickle ends the update grid with RegenCustomMeasurements.RegenMeasKey, RegenMeasurements.Created, RegenMeasurements.Updated, RegenMeasurements.UpdUser, RegenMeasurements.RetrievalID, then the criteria field RegenMeasurements.MeasurementID. Created updates from [CreatedDate], Updated updates from [UpdatedDate], and MeasurementID uses the [RegenMeasurementID] criteria parameter.

Processing Tab

The Processing tab is optional and is available only in Upgrade Project mode. It is greyed out and skipped in New Project and Existing Project modes.

The Processing tab uses a pink accent and contains two independent, default-off import options.

Option 1: Import Processing Tables From Existing Database

Use this option when the old project database contains completed project-specific merchantability standards, volume parameters, regression criteria, or regression equations that should be carried into the upgraded output.

Compatible Four-Table Import

When the old database contains all four Processing setup tables and SpeciesMerchantabilityStandards contains the required fields, Build completely replaces these four output tables:

  • SpeciesMerchantabilityStandards
  • SpeciesNationalVolumeParms
  • SpeciesRegressionCriteria
  • SpeciesRegressions

The required merchantability field names are:

  • HasDRC
  • DMsusceptible
  • DMcollected

Field-type confirmation: Project Pickle checks that these three field names exist. Before relying on the complete four-table copy, also confirm in Microsoft Access Design View that each field uses the expected Yes/No data type.

A compatible import is a complete replacement, not a merge. Rows inherited from the current master database version are deleted from all four selected output tables before rows from the old database are inserted.

Three-Table Fallback For Older Databases

Some older databases either do not contain SpeciesMerchantabilityStandards or contain that table without one or more of the three required fields.

In that situation, Project Pickle displays a warning titled:

Processing import will continue with three tables

Build then completely replaces only these three compatible output tables from the old database:

  • SpeciesNationalVolumeParms
  • SpeciesRegressionCriteria
  • SpeciesRegressions

Project Pickle does not copy the old SpeciesMerchantabilityStandards table during the fallback. The version inherited from the selected current master database remains unchanged in the output.

Manual follow-up is required. After Build, review the upgraded output and manually recreate any old project-specific merchantability standards that still need to be retained.

The fallback deliberately does not invent values for missing merchantability fields and does not attempt to copy only some columns from an incompatible merchantability table.

Tables That Cannot Be Skipped

The following three old-database tables are mandatory whenever the first Processing option is selected:

  • SpeciesNationalVolumeParms
  • SpeciesRegressionCriteria
  • SpeciesRegressions

If any of these three tables is missing, Project Pickle cannot use the fallback. It stops the Processing setup import and reports the missing table. No Processing setup table replacement is started.

For either the four-table plan or three-table plan, Project Pickle writes source and output row counts for every copied table to the detailed run log.

Option 2: Import Plot Summaries, Tree Calc, And Regen Calc Tables

The checkbox Import Plot Summaries, Tree Calc, and Regen Calc tables independently replaces all output rows in:

  • PlotSummaries
  • TreeCalculations
  • RegenCalculations

This is also a complete replacement. An empty source table produces an empty target table. Review the source row counts before using this option.

This second option does not depend on the first Processing option. It can be selected by itself, including when the old SpeciesMerchantabilityStandards table requires the three-table fallback.

Use the second option only when the old calculation and summary rows belong to the plot, tree, and regen inventory data being imported into the upgraded database.

Processing Workflow

  1. Select Upgrade Project mode.
  2. Choose the old database in Existing/old project database.
  3. Choose the current master database version.
  4. Open the Processing tab.
  5. Select Import Processing tables from existing database when the old project's volume and regression setup should be retained.
  6. Select Import Plot Summaries, Tree Calc, and Regen Calc tables when those old calculation and summary rows should replace the current master database version's rows.
  7. Select either option by itself or select both.
  8. Continue to Build.
  9. Read any three-table fallback warning before continuing.
  10. Review the copied table names and source/target row counts in the detailed run log.
  11. Open the output database and verify every selected table.
  12. When the fallback was used, manually review and recreate any project-specific merchantability standards that are still required.

  • Confirm that Tree Remarks still shows TEXT,40,5.
  • In Upgrade Project mode, review whether working AppColumns, completed report tables, old CFIDEER query definitions, Processing setup tables, or Plot/Tree/Regen calculation tables should be copied from the old database.
  • When the Processing fallback warning appears, remember that SpeciesMerchantabilityStandards will remain from the current master and may require manual recreation after Build.
  • Remember that working AppColumns copy, completed Reports copy, Processing copies, calculated-table copy, and old CFIDEER query copy replace complete target sets rather than merging selected rows.

Open Saved Project Database After Build

Check Open saved project database after successful build if you want Access to open the finished saved project database automatically. This saves you from browsing to the Save Project Location folder manually.

Run Log

The Build tab shows progress messages with elapsed time. Project Pickle writes a detailed text log beside the file named in Save Project Location. The file name looks like ProjectName.ProjectPickleRunLog.Success.20260608_123456.txt or ProjectName.ProjectPickleRunLog.Failed.20260608_123456.txt.

Project Pickle writes the detailed text log during build cleanup, either after a successful run or after a handled failure. The Build tab is the live progress view while the app is running.

The text run log includes the run ID, step number, log time, run status, step category, project mode, project name, Existing/old project database when used, current master database version when used, saved project database, and message. It also includes timing fields: ElapsedSeconds, StepDurationSeconds, ElapsedTime, and StepDuration. Use StepDurationSeconds to find which step took the longest. Project Pickle writes a final total run-time message before it exports the log.

The detailed text log is written even when the build fails, as long as Project Pickle can write to the output folder or the Project Pickle App Files support folder. Use this file when troubleshooting because it can survive an Access table-write failure.

During upgrade builds, the log also includes diagnostic checks after import, moved-field mapping, and repair. These checks compare old and output row counts, compare measurement rows by period, check whether plot/tree/regen measurement rows have matching custom measurement rows, and check whether measurement rows have matching parent plot or tree rows. If a check finds missing links, Project Pickle writes sample IDs to the log so those records are easier to find.

During the upgrade repair step, Project Pickle now logs each table it is checking. For large tables, it reports progress every few hundred rows so users can tell the app is still working.

If The App Looks Frozen

Large upgrades can take time, especially when Project Pickle is importing many plot, tree, regen, and assignment rows. Project Pickle refreshes the Build tab at log points, but a single large Access write may still briefly make Windows show Not Responding. Watch the Build tab log. During upgrade imports, Project Pickle shows each large table copy with the number of rows copied, elapsed time, and rows per second. If messages are still appearing, Project Pickle is still working. If nothing changes for a long time, wait a few more minutes before closing because Access writes can be slow on large databases.

If you try to close the app while Project Pickle is processing, Project Pickle warns you first. Choosing Yes requests cancel and closes the app after the current safe checkpoint. Closing during a database write can leave the output copy incomplete.

Common Workflows

New Project With No Plot Data Yet

  1. Mode: New Project.
  2. Select the current master database version. The Existing/old project database box stays greyed out.
  3. Choose a Save Project Location name.
  4. Enter project name, measurement cycle interval, region, agency, and reservation.
  5. Add the first period on the Periods tab.
  6. Leave the Plots tab blank.
  7. Skip assignment tabs.
  8. Review AppColumns only if needed now.
  9. Leave CFIDEER queries off unless fields are final enough for data entry.
  10. Build.

New Project With A Plot List

  1. Complete the Project and Periods tabs.
  2. On the Plots tab, choose the spreadsheet.
  3. Click Load columns and map the spreadsheet columns.
  4. Use Plot Assignments if assignment setup is ready.
  5. Build.

Existing Project Update

  1. Mode: Existing Project.
  2. Select the current-format project database in Existing/old project database.
  3. Choose a new Save Project Location name.
  4. Click Load lists.
  5. Leave the Periods tab read-only. If a new measurement period still needs to be added, stop and use New Project or Upgrade Project mode instead.
  6. Use AppColumns if you need to add new field data items.
  7. Use the Plots tab if you need to add new plots. Project Pickle skips plots that already exist by PlotNumber and adds only new ones.
  8. Review Get Field Order and CFIDEER Queries if field order or query setup changed.
  9. Use Plot Assignments when assignment tables need to be rebuilt for the current period.
  10. Make only the other updates needed, such as reports or validation.
  11. Build.

Upgrade Project To A Clean Newer Template

  1. Mode: Upgrade Project.
  2. Select the old project database in Existing/old project database.
  3. Select the current master database version.
  4. Choose a new Save Project Location name.
  5. Load Region, Agency, Reservation, and periods from the old project.
  6. Add the new measurement period for the remeasurement and mark it current.
  7. Review AppColumns. Either rebuild the AppColumns setup or select complete AppColumns copy.
  8. Use Upgrade Mapping for old fields that need to move, be renamed, or remain as historical custom fields.
  9. Confirm old inventory import is checked when plot, tree, regen, and assignment data should be migrated.
  10. Choose one CFIDEER approach:
    • Load Get Field Order and rebuild the queries, or
    • Check Copy existing _prjCFIDEER_ queries from old database and skip Get Field Order/query rebuilding.
  11. On Processing, independently decide whether to copy:
    • The four Processing setup tables, and/or
    • PlotSummaries, TreeCalculations, and RegenCalculations.
  12. For the first Processing option, confirm SpeciesMerchantabilityStandards contains HasDRC, DMsusceptible, and DMcollected.
  13. Review Reports and assignments.
  14. Build.
  15. Review the detailed run log and verify the output database in Access.

Remeasurement With A New Period

  1. Use Upgrade Project mode whenever a new measurement period must be added.
  2. Select the old project database in Existing/old project database.
  3. Select the current master database version.
  4. Choose a new Save Project Location.
  5. Load the existing periods from the old project database.
  6. Add the new measurement period and mark it current.
  7. Review AppColumns and Upgrade Mapping.
  8. Choose one CFIDEER approach:
    • Load Get Field Order and rebuild the queries, or
    • Copy the existing _prjCFIDEER_ queries from the old database.
  9. Review Reports, assignments, and both Processing options.
  10. Build and verify the new period and related measurement rows in the output database.

Existing Project mode cannot add a measurement period. Existing Project preserves the period rows already present in the selected current-format database. Use it only after the required period structure already exists.

After The Build: What To Check In Access

After Project Pickle finishes, open the saved project database and review the results before production use.

  • Confirm the project name and ProjectID are correct.
  • Confirm region, agency, and reservation values are correct.
  • Confirm measurement periods are correct.
  • Confirm plots were imported if you used plot import.
  • Confirm blank plot measurement rows were created for the new period if expected.
  • Confirm matching custom measurement rows exist where expected.
  • Confirm AppColumns active/inactive settings are correct.
  • Confirm AppColumnsArchive exists if you want a before-build snapshot of the original AppColumns table.
  • Confirm AppColumnCodes has one row per code.
  • Confirm report headers and report columns were updated if that option was used.
  • Confirm validation rules have the correct ProjectID and applicable periods if that option was used.
  • Confirm CFIDEER queries exist and look correct after the build.
  • Open the detailed text run log beside the saved project database and review the saved run messages.
  • If Processing setup copy was selected, confirm row counts and values in SpeciesMerchantabilityStandards, SpeciesNationalVolumeParms, SpeciesRegressionCriteria, and SpeciesRegressions.
  • If calculated/summary copy was selected, confirm row counts and record links in PlotSummaries, TreeCalculations, and RegenCalculations.
  • If old CFIDEER query copy was selected, confirm the expected _prjCFIDEER_ queries exist and open correctly.
  • Confirm the run log says either that CFIDEER queries were copied or that they were rebuilt, according to the option selected.

Troubleshooting

Project Pickle Says A Required Entry Is Missing

Read the message and return to the named tab. Most missing-entry messages mean Project Pickle has not started changing the saved project database yet.

Region, Agency, Or Reservation Is Blank

  • Make sure the database path required by the selected mode is filled in.
  • Click Load lists again.
  • If using New Project mode, make sure Current master database version is selected.
  • If using Upgrade or Existing mode, make sure Existing/old project database is selected.

Periods Did Not Load

  • Make sure the selected database is the project database that already has period rows.
  • In Existing Project mode, the Periods tab is read-only and Project Pickle preserves the period rows already in the selected database during build.
  • In Upgrade Project mode, click Load periods from database on the Periods tab if you need to reload period rows from the old project database.
  • If no periods are found, add the needed periods manually.

AppColumns Did Not Load

  • Load Standard AppColumn fills the top review grid from the current master/current-format AppColumns table.
  • There is no separate Existing AppColumn list. Existing/project rows show in the same top grid only after they already exist in the selected database.
  • This usually means custom AppColumns were added during a previous Upgrade Project or Existing Project run.
  • Make sure the correct database path is selected before clicking Load Standard AppColumn.

AppColumns Copy Mode Is Locked

When Copy AppColumns from existing database is checked, manual AppColumns loading, editing, code-list editing, new custom rows, and Copy selected format are intentionally disabled. Uncheck the copy option to return to manual AppColumns setup.

The option is valid only in Upgrade Project or Existing Project mode. The source database must contain both AppColumns and AppColumnCodes.

A Reports Layout Will Not Validate

  • Load the database report layouts before adding custom fields.
  • Give every report-column row a FieldFormat.
  • Remove duplicate fields.
  • Keep custom fields before RemarksPlot, RemarksTree, or RemarksRegen.
  • Do not remove the required remarks field.
  • Use spaces instead of tabs.
  • Keep every nonblank header line the same length.
  • Keep each header line at or below 255 characters.
  • Use printable ASCII characters in fixed-width headers.

When completed report copy is checked in Upgrade Project mode, the manual report controls are intentionally disabled. Uncheck the copy option to return to manual layout editing.

Completed Report Copy Cannot Resolve A Field

The old ReportColumns table references an AppColumns field that is not present in the output database. Select Copy AppColumns from existing database too, or create a matching AppColumns row with the same App table and field name before copying the reports.

Processing Import Was Turned Off

Project Pickle turns off the first Processing option when the selected old database cannot safely provide the complete four-table Processing setup.

  • Make sure the app is in Upgrade Project mode.
  • Make sure the correct old database is selected.
  • Confirm both source and target contain: SpeciesMerchantabilityStandards, SpeciesNationalVolumeParms, SpeciesRegressionCriteria, and SpeciesRegressions.
  • Confirm the old SpeciesMerchantabilityStandards table contains HasDRC, DMsusceptible, and DMcollected.
  • Do not add empty fields only to bypass the check.
  • Choose a compatible old database, or leave the first option off so the Processing setup from the selected current master database version remains.
  • The second Plot Summaries/Tree Calc/Regen Calc option can still be used independently when its three source tables exist.

CFIDEER Query Copy Is Disabled

  • The old-query copy checkbox is available only in Upgrade Project mode.
  • It remains off and disabled in New Project and Existing Project mode.
  • When checked, it intentionally disables Get Field Order and the normal query-rebuild checkbox.
  • When unchecked, Get Field Order and normal query rebuilding become available again.

CFIDEER Query Rebuild Failed After Get Field Order Was Skipped

When the normal query-rebuild option is checked, return to Get Field Order and click Load fields from AppColumns setup. Review the Plot, Tree, and Regen lists, then build again. Get Field Order can be skipped only when old query-copy mode is selected.

Quick Checklist

Use this short checklist when you just need a fast reminder.

  1. Pick the correct mode.
  2. New Project: select the current master database version.
  3. Upgrade Project: select both the old project database and the current master database version.
  4. Existing Project: select the current-format project database in Existing/old project database.
  5. Choose a clear Save Project Location name.
  6. Load or confirm Region, Agency, and Reservation information.
  7. For New or Upgrade mode, enter or load measurement periods and mark one period current. Existing Project keeps its existing periods.
  8. Use Plots only when new plot data is ready.
  9. Use Plot Assignments only when plots exist.
  10. Use Tree Assignments only in Upgrade Project mode with tree rows.
  11. Review AppColumns and add custom fields, or use complete AppColumns copy when appropriate.
  12. Use Upgrade Mapping when old history fields need to be preserved, renamed, or moved.
  13. Choose one CFIDEER path:
    • Load Get Field Order and rebuild the queries, or
    • In Upgrade Project mode, copy the existing _prjCFIDEER_ queries from the old database.
  14. Review Reports and save/validate manual layouts when report fields or formatting need to change.
  15. In Upgrade Project mode, decide whether to copy completed report tables.
  16. In Upgrade Project mode, independently decide whether to copy:
    • The four Processing setup tables, and/or
    • PlotSummaries, TreeCalculations, and RegenCalculations.
  17. For the first Processing option, confirm SpeciesMerchantabilityStandards contains HasDRC, DMsusceptible, and DMcollected.
  18. Build, review the detailed run log, and verify the saved database in Microsoft Access.
